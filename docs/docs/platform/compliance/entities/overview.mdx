# Entities

Entities represent business units, subsidiaries, departments, or other organizational components that require separate compliance management and oversight within Openlane.

## What Are Entities?

Entities are organizational units or business components that have distinct compliance requirements, operational boundaries, or regulatory obligations. They enable organizations to manage compliance at granular levels while maintaining centralized oversight and coordination.

## Compliance Significance

Entities are essential for:
- **Regulatory Compliance**: Meeting jurisdiction-specific requirements for subsidiaries
- **Organizational Structure**: Mapping compliance responsibilities to business units
- **Risk Management**: Isolating and managing risks at the entity level
- **Audit Scope**: Defining audit boundaries and responsibilities
- **Resource Allocation**: Assigning compliance resources to specific entities

## Entity Types

### Legal Entities
- **Definition**: Legally distinct organizations (subsidiaries, joint ventures)
- **Compliance Scope**: Full regulatory compliance responsibilities
- **Examples**: Regional subsidiaries, acquired companies, partnerships
- **Characteristics**: Separate legal identity, distinct regulatory requirements

### Business Units
- **Definition**: Operational divisions within a legal entity
- **Compliance Scope**: Operational compliance and control implementation
- **Examples**: Product divisions, regional offices, functional departments
- **Characteristics**: Shared legal identity, specialized compliance needs

### Functional Entities
- **Definition**: Cross-functional groups with specific compliance roles
- **Compliance Scope**: Specialized compliance activities and controls
- **Examples**: IT security team, compliance department, audit committee
- **Characteristics**: Specialized expertise, cross-organizational responsibilities

### Geographic Entities
- **Definition**: Location-based organizational units
- **Compliance Scope**: Jurisdiction-specific compliance requirements
- **Examples**: Regional offices, country operations, data centers
- **Characteristics**: Geographic boundaries, local regulatory requirements

## Properties

### Core Information
- **ID**: Unique identifier for the entity
- **Name**: Official entity name
- **Display Name**: User-friendly name for the entity
- **Description**: Purpose and scope of the entity
- **Entity Type**: Classification of the entity (legal, business unit, etc.)

### Organizational Structure
- **Parent Entity**: Hierarchical relationship to parent entities
- **Child Entities**: Subordinate entities under this entity
- **Organization**: Parent organization that owns the entity
- **Status**: Current operational status (active, inactive, merged)

### Geographic and Operational
- **Domains**: Associated domains and web properties
- **Locations**: Physical locations and addresses
- **Jurisdictions**: Applicable legal jurisdictions
- **Operational Scope**: Areas of business operation

### Compliance Integration
- **Contacts**: Associated personnel and contact information
- **Assets**: IT assets and infrastructure associated with the entity
- **Documents**: Relevant documentation and records
- **Scans**: Security and compliance scans performed on entity assets

## API Access

Entities are managed exclusively through GraphQL operations. There are no REST endpoints for entity CRUD operations.

## Security and Compliance Considerations

### Access Control
- **Role-Based Access**: Granular permissions for entity management
- **Hierarchical Permissions**: Inherited permissions from parent entities
- **Cross-Entity Access**: Controlled access across entity boundaries
- **Audit Permissions**: Specialized permissions for compliance auditing

### Data Protection
- **Data Residency**: Compliance with data residency requirements by entity
- **Privacy Regulations**: Entity-specific privacy regulation compliance
- **Data Classification**: Classification of data associated with entities
- **Cross-Border Transfer**: Management of data transfers between entities

### Compliance Monitoring
- **Entity-Level Reporting**: Compliance reporting at the entity level
- **Audit Scope**: Definition of audit boundaries and responsibilities
- **Risk Assessment**: Entity-specific risk assessments and management
- **Control Implementation**: Entity-specific control implementation and monitoring

## Entity Lifecycle Management

### Entity Creation
```javascript
// Create new entity with compliance setup using GraphQL
const createEntity = async (entityData) => {
  const mutation = `
    mutation CreateEntity($input: CreateEntityInput!) {
      createEntity(input: $input) {
        entity {
          id
          name
          displayName
          description
          domains
          createdAt
        }
      }
    }
  `;

  const variables = {
    input: {
      name: entityData.name,
      displayName: entityData.displayName,
      description: entityData.description,
      domains: entityData.domains
    }
  };

  const response = await fetch('https://api.openlane.io/query', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${apiToken}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      query: mutation,
      variables: variables
    })
  });

  const result = await response.json();
  return result.data.createEntity.entity;
};
```

## Common Use Cases

### Multi-National Compliance
```yaml
# Entity structure for multi-national organization
entities:
  - name: "ACME Corp"
    type: "parent"
    jurisdiction: "US"
    children:
      - name: "ACME Europe GmbH"
        type: "subsidiary"
        jurisdiction: "DE"
        compliance_frameworks: ["GDPR", "ISO 27001"]
      - name: "ACME Asia Pte Ltd"
        type: "subsidiary"
        jurisdiction: "SG"
        compliance_frameworks: ["PDPA", "ISO 27001"]
```

### Departmental Compliance
```yaml
# Entity structure for departmental compliance
entities:
  - name: "IT Department"
    type: "functional"
    responsibilities:
      - "Infrastructure security"
      - "Data protection"
      - "System compliance"
    assets:
      - "Production servers"
      - "Development environments"
      - "Network infrastructure"
```

### Merger and Acquisition
```javascript
// Handle entity integration during M&A
const integrateAcquiredEntity = async (acquiredEntityData) => {
  // Create new entity for acquired company
  const newEntity = await createEntity({
    name: acquiredEntityData.name,
    displayName: acquiredEntityData.displayName,
    description: `Acquired entity - ${acquiredEntityData.name}`,
    entityType: 'acquired-entity',
    domains: acquiredEntityData.domains,
    status: 'integration'
  });

  // Migrate existing compliance data
  await migrateComplianceData(acquiredEntityData.id, newEntity.id);

  // Update status after integration
  await updateEntityStatus(newEntity.id, 'active');
};
```

## Relationships

Entities integrate with several other Openlane objects:

### Direct Relationships
- **Entity Type**: Classification and template for entity configuration
- **Organization**: Parent organization that owns the entity
- **Contacts**: Personnel associated with the entity
- **Assets**: IT assets and infrastructure owned by the entity

### Indirect Relationships
- **Controls**: Entity-specific control implementations
- **Evidence**: Evidence associated with entity operations
- **Risks**: Entity-specific risk assessments and management
- **Policies**: Entity-specific policies and procedures
- **Tasks**: Entity-specific compliance tasks and assignments

## Best Practices

### Entity Design
1. **Clear Boundaries**: Define clear operational and compliance boundaries
2. **Consistent Naming**: Use consistent naming conventions across entities
3. **Hierarchical Structure**: Maintain logical hierarchical relationships
4. **Purpose Definition**: Clearly define the purpose and scope of each entity

### Compliance Management
1. **Jurisdiction Mapping**: Map entities to appropriate legal jurisdictions
2. **Regulatory Alignment**: Align entity structure with regulatory requirements
3. **Audit Planning**: Consider entity structure in audit planning and execution
4. **Resource Allocation**: Allocate compliance resources based on entity needs

### Operational Considerations
1. **Asset Management**: Properly associate assets with appropriate entities
2. **Contact Management**: Maintain accurate contact information for each entity
3. **Status Tracking**: Keep entity status current and accurate
4. **Documentation**: Maintain comprehensive documentation for each entity

### Integration Guidelines
1. **Data Consistency**: Ensure data consistency across related entities
2. **Access Control**: Implement appropriate access controls for entity data
3. **Reporting**: Design reporting structures that respect entity boundaries
4. **Audit Trail**: Maintain complete audit trails for entity operations

Entities provide the organizational structure necessary for effective compliance management in complex organizational environments.
